# Performance Audit

Kotlin/Spring Boot 프로젝트의 성능 병목을 체계적으로 감사한다.

## 감사 절차

### Step 1: 데이터베이스 접근 패턴

1. Repository 인터페이스와 커스텀 쿼리를 분석한다
2. N+1 쿼리 패턴을 탐지한다:
   - `@OneToMany`, `@ManyToOne` 관계에서 Lazy Loading 혹은 루프 조합
   - Fetch Join 미사용
3. 불필요한 전체 조회 (`findAll` 무조건 사용)를 탐지한다
4. 인덱스가 필요한 쿼리 조건을 식별한다

### Step 2: 동시성 제어

쿠폰 발급 등 동시성이 중요한 로직을 분석한다:
- 비관적 락 vs 낙관적 락 선택의 적정성
- 락 범위가 과도하게 넓은 경우 (테이블 락 vs 행 락)
- Redis 분산 락 사용 시 TTL, 재시도 전략
- 데드락 가능성이 있는 락 순서

### Step 3: 캐싱 전략

Redis 캐싱 사용 패턴을 분석한다:
- 캐시 적중률에 영향을 주는 키 설계
- TTL 설정의 적정성
- 캐시 무효화 전략 (데이터 일관성 vs 성능)
- 직렬화/역직렬화 비용 (Jackson 설정)

### Step 4: 트랜잭션 관리

- `@Transactional` 범위가 과도하게 넓은 경우
- 읽기 전용 작업에 `readOnly = true` 미설정
- 트랜잭션 내 외부 호출 (HTTP, Redis) 포함 여부
- 트랜잭션 전파 수준의 적정성

### Step 5: 리소스 사용

- 불필요한 객체 생성 패턴 (루프 내 객체 생성 등)
- 스트림/시퀀스 미활용 (대량 데이터 처리 시)
- 커넥션 풀 설정 적정성
- 비동기 처리가 필요한 동기 작업

### Step 6: 부하 테스트 정합성

k6 테스트 시나리오가 있으면 분석한다:
- 테스트 시나리오가 실제 사용 패턴을 반영하는가?
- 테스트에서 측정하는 지표와 코드의 관계

## 심각도 기준

| 심각도 | 기준 |
|--------|------|
| Critical | N+1 쿼리, 데드락 가능성, 과도한 트랜잭션 범위 |
| Warning | 캐시 미활용, readOnly 미설정, 비효율적 직렬화 |
| Info | 최적화 가능하지만 현재 부하에서 문제 없는 사항 |
