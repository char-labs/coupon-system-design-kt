# Review Checklist

## 목적

이 문서는 `technical-writing-reviewer` 가 확인해야 하는 항목을 정리한다.

## 1. 사실 검증

- `POST /coupon-issues` 의 `SUCCESS` 를 최종 DB 발급 완료처럼 썼는가
- outbox 를 intake durability 로 설명했는가
- `request table`, `relay`, `CDC`, durable acceptance 를 이미 구현된 것처럼 썼는가
- worker component scan 예외를 숨겼는가
- Redis Lua, Redisson lock, RRateLimiter 책임이 섞였는가
- Kafka, Redis, DB 중 최종 truth 의 위치를 잘못 설명했는가

## 2. Reference 품질

- 외부 `근거 팩` 이 있는가
- `근거 팩` 이 `External References` 중심으로 정리되어 있는가
- 외부 근거가 최소 2개 이상 있는가
- 공식 문서보다 2차 블로그에 과도하게 기대지 않았는가
- benchmark 블로그가 구조 참고로만 쓰였는가
- `Writing Benchmarks` 가 없을 때도 충분하면 굳이 추가하라고 하지 않는가
- benchmark 항목마다 구조 영향 이유를 짧게 설명할 수 있는가
- 주제와 직접 관련 없는 benchmark 링크가 관성적으로 들어가 있지 않은가
- 본문의 핵심 주장과 reference 가 실제로 연결되는가
- publish용 `초안` 에 raw `Local References` 섹션이 남아 있지 않은가
- 로컬 근거가 코드 컨텍스트 블록, 짧은 코드 발췌, 흐름 설명으로 자연스럽게 흡수되었는가

## 3. 이미지 제안

- frontmatter 아래 대표 이미지가 있다면 실제 주제와 맞는가
- 대표 이미지가 없다면 굳이 넣지 않아도 되는 글에 억지로 강제하지 않았는가
- 이미지 제안이 있는가
- 글로만 충분한 구간에 이미지를 과하게 넣지 않았는가
- 비동기 경계, 대안 비교, 상태 전이 같은 구간에 필요한 이미지가 빠지지 않았는가
- 이미지 유형과 캡션이 섹션 논리에 맞는가
- Mermaid 라벨이 raw 명령명 나열보다 독자 친화적인 한국어 의미 설명으로 되어 있는가
- raw command name 이 꼭 필요한 경우가 아니라면 코드 블록/본문으로 내리고, 다이어그램은 의미 중심으로 정리했는가

## 4. 글 구조

- MDX 초안이라면 frontmatter가 있는가
- frontmatter의 title/description/date/tags가 글 내용과 맞는가
- `서론` 으로 시작하는 출판 하네스가 자연스러운가
- 질문형 도입이 있는가
- 초반에 `왜 이 글을 쓰는지` 가 보이는가
- 대안과 현재 선택이 모두 보이는가
- Redis Lua 주제라면 `reserve / release / rebuild` 가 각각 설명되어 있는가
- `왜 이 선택이 맞는가` 와 `한계` 가 함께 있는가
- 필요하면 `예상치 못한 이슈`, `헷갈리기 쉬운 지점`, `후기` 같은 환기용 섹션이 있는가
- 문단이 너무 길지 않은가
- 목차나 소제목이 읽기 흐름을 돕는가
- 마무리에 교훈 또는 다음 단계가 있는가
- 핵심만 남긴다고 설명이 과하게 압축되지는 않았는가
- reader 가 추가 질문할 만한 핵심 경계나 트레이드오프가 빠지지 않았는가
- 어려운 구간에서 최소 한 단계 더 풀어쓴 설명이 있는가

## 5. 헤드라인과 문장 톤

- 제목만 읽어도 어떤 기술과 맥락을 다루는지 보이는가
- description 만 읽어도 왜 이 기술을 선택했는지가 보이는가
- 제목이 구체 기술명 + 맥락 + `사용기/과정/에 대해/왜` 패턴 중 하나를 따르는가
- 과한 카피 문구나 낚시성 표현이 없는가
- 본문 주요 서술이 `합니다체` 인가
- 직접 인용이나 정의문이 아닌데 `한다체` 가 반복되지 않는가
- `백엔드 개발자 관점에서` 같은 메타 독자 프레이밍보다 직접 경험형 1인칭 서술이 자연스럽게 쓰였는가

## 5-1. Reference Noise Gate

- inline citation anchor를 쓴 경우 `href="#ref-..."` 와 최종 `References` 의 `id` 가 맞는가
- inline anchor가 없더라도 출판 플랫폼에서 깨질 HTML을 무분별하게 넣지 않았는가
- benchmark section 이 있다면, 실제 본문 구조에 보이는 흔적이 있는가
- benchmark section 이 없어도 글의 완성도가 충분하다면 생략을 권고했는가
- 예를 들어 monorepo 글을 Redis Lua 글에 붙이는 식의 topic mismatch 를 노이즈로 판단했는가
- `이 링크가 왜 여기 있는지 한 문장으로 설명 가능한가` 를 reviewer 가 직접 점검했는가

## 6. 개인 문체와 비유 사용

- 작성자의 기존 블로그 톤처럼 너무 딱딱하지 않은가
- 친근한 1인칭 설명이 있어도 기술 정확도가 흐려지지 않는가
- 비유가 1~2개 수준으로 절제되어 있는가
- 비유가 웃기기만 하고 설명에는 도움이 안 되는 상태는 아닌가
- 비유 다음에 기술 설명으로 제대로 회수했는가

## 7. Reviewer 출력 형식

Reviewer 는 가능하면 아래 형식으로 반환한다.

```md
치명적 사실 오류:
- ...

보강 필요:
- ...

이미지 제안 조정:
- ...

레퍼런스 누락 또는 약한 근거:
- ...

최종 반영 메모:
- ...
```

## 8. Agent 재협업 루프

- reviewer 가 material issue 를 찾았을 때 drafter 에게 바로 반영 가능한 수정 delta 를 남겼는가
- 수정 delta 가 제목, 구조, 사실 오류, 코드 컨텍스트, 이미지, 근거 팩 중 어디를 고쳐야 하는지 분명한가
- redraft 뒤 reviewer 가 확인 pass 를 다시 했는가
- loop 가 무한 반복되지 않고 기본 한 번의 재작성으로 정리되었는가
