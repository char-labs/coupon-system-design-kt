## 목적

이 문서는 기술 사실의 근거가 아니라, 글의 구조와 전달 방식을 참고하기 위한 벤치마크를 정리한다.

### 관찰한 패턴

- 강한 부제가 바로 붙는다
- 초반에 목차를 명시한다
- 문제를 `사례 1 / 사례 2` 처럼 나눠서 설명한다
- 비교 후보를 표로 보여준다
- `최종 결정` 섹션을 따로 둔다
- 마지막에 `교훈` 또는 `영향도 평가 기준`처럼 정리된 결론을 준다

### 이 저장소 글에 적용할 규칙

- 질문형 글이라도 목차를 초반에 붙일 수 있다
- `대안 비교표` 는 가능한 한 한 번 넣는다
- `최종 결정` 또는 `왜 지금은 이 선택이 맞는가` 섹션을 둔다

## Velog 패턴

### 수집 정책

- 좋아요 30개 이상 글을 우선한다
- 좋아요 수가 검색 인덱스에 안정적으로 노출되지 않으면 `velog` 트렌딩/추천 노출 글로 대체한다
- 이 기준은 사실의 근거가 아니라 `고반응 글의 구조적 특징` 을 모으기 위한 정책이다

### 관찰한 패턴

- 초반에 결과나 핵심 결론을 미리 준다
- 질문형 소제목을 자주 쓴다
- 문단이 짧다
- 숫자 기반 섹션 구조가 많다
- 실전 경험과 시행착오를 앞단에 둔다

### 이 저장소 글에 적용할 규칙

- 서론 2~3문단 안에 핵심 결론 힌트를 준다
- 긴 문단보다 짧은 문단을 우선한다
- 소제목은 설명형보다 질문형을 우선 검토한다

## Author Blog 패턴

### 참고 채널

- [ybchar.dev Tech Blog](https://ybchar.dev/tech-blog)
- [yb__char velog](https://velog.io/@uiurihappy)
- [ybchar.dev 메인 소개](https://www.ybchar.dev/)

### 최근 기술 글에서 본 제목 패턴

- 구체 기술명 + 맥락 + `사용기` 또는 `과정`
- 짧은 훅 문구 + 쉼표 + 주제 + `에 대해`
- 제목만 봐도 다루는 기술과 설명 범위가 드러난다
- 추상적인 카피보다 `무엇을, 왜, 어떤 맥락에서` 다루는지가 제목에 들어간다

예시:

- `LocalStack과 DynamoDB 사용기`
- `Java 21, Spring boot 3.4.x 마이그레이션 과정`
- `소프트웨어 디자인도 전략적으로, 전략 패턴에 대해`

### 최근 기술 글에서 본 본문 패턴

- 첫 문단에서 작성 계기나 문제 상황을 바로 깐다
- 개념 설명 전에 경험을 먼저 깔아 독자를 끌어당긴다
- `서론/계기 -> 과정/구조 설명 -> 예상치 못한 이슈 -> 후기` 흐름이 자주 보인다
- 개념 설명과 예시 코드, 이미지를 섞는다
- 중간에 `왜 갑자기 ...?`, `Spring에서는 어떻게?` 같은 질문형 소제목을 둔다
- 기술 설명이 길어질 때는 `간단한 확장 예시`, `예상치 못한 예외 발견` 같은 환기용 제목을 넣는다

### 최근 기술 글에서 본 문체 패턴

- 기본 종결은 `합니다`, `해보겠습니다`, `봤습니다` 계열이다
- 아주 가끔 구어체나 장난기가 섞여도 전체 문장은 정중한 서술체를 유지한다
- 너무 차갑지 않은 1인칭 실무 공유 톤이 중심이다
- `이 경험을 공유하고자`, `한번 정리해보겠습니다` 같은 연결 문장이 자연스럽다

### 적용 규칙

- 서론은 `이 문제가 왜 거슬렸는지` 또는 `어디서 헷갈리기 쉬운지` 로 시작한다
- 서론 첫 2~3문단 안에 `왜 이 글을 쓰는지` 를 넣는다
- 실무에서 바로 연결되는 한 문장을 초반에 둔다
- 제목은 가능한 한 구체 기술명과 설명 범위를 함께 드러낸다
- 기술 글 기본 종결은 `합니다체` 로 고정한다
- 설명 중간에 짧은 비유를 1~2개 정도만 허용한다
- 비유는 `웃기기 위한 장치` 가 아니라 `이해를 빠르게 만드는 장치` 로 쓴다
- 어조는 친근하게 가져가되 기술 근거와 구조 설명은 흐리지 않는다

## 하이브리드 패턴

최종 글은 아래 조합을 기본형으로 쓴다.

- 도입: 무신사식 질문형 시작
- 구조: 올리브영식 목차와 비교표
- 본문 길이감: velog 식 짧은 문단
- 문체: author blog 식 친근한 1인칭 실무 톤 + `합니다체`
- 시각자료: 무신사식 그림 번호와 다이어그램
- 결론: 올리브영식 `최종 결정 / 교훈`
- 제목: author blog 식 구체 기술명 + 과정/사용기 또는 훅 + `에 대해`

## 금지 규칙

- 특정 블로그의 말투를 그대로 복제하지 않는다
- 벤치마크 블로그를 사실의 근거로 인용하지 않는다
- 구조만 따라가고 논지가 비어 있는 글을 만들지 않는다
- 제목만 과하게 자극적으로 만들고 본문은 평이한 설명으로 끝내지 않는다
