## 현재 확인 가능한 스타일 신호

현재 접근 가능한 공개 페이지 기준으로 아래 성향이 보인다.

- 인삿말과 자기소개가 비교적 친근하다
- 성장, 적응, 개선, pain point 해결을 중요하게 둔다
- 레거시 개선이나 실무 문제 해결에 보람을 느끼는 관점이 분명하다
- 과하게 포장된 마케팅 문체보다 담백한 실무형 설명이 더 잘 맞는다
- 완전히 무미건조한 교과서 톤보다는 약간 사람 냄새 나는 문장이 어울린다

## 글에 반영할 톤

- 기본은 기술적으로 단단한 설명
- 서론과 전환 문단은 조금 더 친근하게
- `내가 이 문제를 이렇게 봤다` 는 1인칭 관찰 톤 허용
- 실무에서 왜 중요한지, 어디가 pain point 인지 먼저 보여주기
- 지나치게 감성적이거나 과장된 카피는 피하기

## 비유 사용 규칙

- 비유는 글당 1~2개 정도만 사용한다
- 비유는 기술 설명의 보조 장치로만 쓴다
- 비유 직후에는 반드시 실제 구조나 동작 설명으로 돌아온다
- 억지로 웃기려는 문장, 밈, 과한 드립은 피한다

## 안전한 비유 예시

- outbox:
  - `메인 업무가 끝난 뒤 영수증을 별도 보관함에 넣어 후속 처리를 맡기는 구조`
- Lua reserve:
  - `입구에서 번호표와 좌석 수를 동시에 확인하는 안내 데스크`
- Kafka intake:
  - `주문 접수와 실제 조리를 같은 테이블에서 하지 않는 구조`
- distributed lock:
  - `작업대에 동시에 두 사람이 올라오지 못하게 막는 안전 바`

## 금지 규칙

- 비유만 기억나고 기술 내용은 남지 않는 글
- 친근함을 살리겠다고 정확성을 희생하는 글
- 작성자 말투를 과장되게 흉내 내는 글
- 설명보다 감탄사와 수식어가 더 많은 글
