# Author Style Profile

## 목적

이 문서는 작성자의 최근 블로그 톤과 헤드라인 패턴을 `coupon-technical-writing` 에 반영하기 위한 기준입니다.

## 최근 분석 대상

최근 작성 글 중 기술 글 성격이 강한 아래 글을 우선 기준으로 삼습니다.

- `2025-03-16` 소프트웨어 디자인도 전략적으로, 전략 패턴에 대해
- `2025-03-02` LocalStack과 DynamoDB 사용기
- `2025-01-01` Java 21, Spring boot 3.4.x 마이그레이션 과정

회고 글은 문장의 온도와 친근한 전환 표현만 약하게 참고합니다.

- `2025-12-29` 2025년은
- `2025-01-19` 2024년 회고 1편
- `2025-01-28` 2024년 회고 2편

## 최근 글에서 읽히는 스타일 신호

최근 글 기준으로 아래 성향이 분명합니다.

- 도입에서 `왜 이 글을 쓰는지`, `어떤 문제가 있었는지` 를 먼저 꺼냅니다
- 경험담을 바로 깔고, 그 뒤에 개념 설명과 코드 설명을 붙입니다
- `예상치 못한 예외`, `후기`, `간단한 확장 예시` 처럼 중간중간 독자의 집중을 환기하는 소제목을 씁니다
- 완전히 딱딱한 논문체보다 친근한 실무 공유 톤이 잘 맞습니다
- 다만 핵심 설명부는 결국 구조와 동작 설명으로 회수합니다
- Pain point, 시행착오, 개선 이유를 숨기지 않습니다

## 문장 종결 규칙

- 기본 서술은 `합니다`, `했습니다`, `해보겠습니다`, `보겠습니다` 계열로 통일합니다
- `한다`, `된다`, `였다` 같은 평서체는 정의문, 코드 주석, 표 항목, 직접 인용이 아니면 피합니다
- 설명형 정의도 `전략 패턴입니다`, `이 구조는 ... 담당합니다` 처럼 정리합니다
- 문장 온도는 친근할 수 있지만 종결은 정제된 `합니다체` 를 유지합니다

## 글에 반영할 톤과 결

- 기본은 기술적으로 단단한 설명입니다
- 첫 2~3문단은 문제 공감이나 계기 설명으로 부드럽게 엽니다
- `제가 이 문제를 겪으면서`, `여기서 헷갈리기 쉬웠던 점은` 같은 1인칭 관찰 톤을 허용합니다
- 독자를 메타로 호출하는 `백엔드 개발자 관점에서`, `독자 입장에서` 같은 표현보다, 작성자가 직접 겪은 상황을 설명하는 문장을 우선합니다
- 실무에서 왜 중요한지, 어디가 pain point 였는지를 먼저 보여줍니다
- 과하게 포장된 마케팅 문체보다 담백한 경험 공유 문체를 우선합니다
- 웃기려는 문장보다 `아, 이 포인트 때문에 썼구나` 가 먼저 느껴지게 씁니다
- 짧게 끝내는 것보다 독자가 이해할 만큼 설명하는 것을 우선합니다
- 특히 최근 기술 글처럼 과정, 예외, 배경 설명은 필요하면 충분히 길게 가져갑니다
- 다만 장황하게 반복하지 말고, `왜`, `어떻게`, `그래서 무엇이 달라지는지` 를 채우는 방향으로 길어져야 합니다

## 헤드라인 패턴

최근 글 제목은 아래 세 패턴이 반복됩니다.

1. 구체 기술명 + 사용기/과정
   - `LocalStack과 DynamoDB 사용기`
   - `Java 21, Spring boot 3.4.x 마이그레이션 과정`
2. 짧은 훅 + 쉼표 + 주제 + `에 대해`
   - `소프트웨어 디자인도 전략적으로, 전략 패턴에 대해`
3. 연도/시기 + 짧은 단문
   - 회고 글에서 사용

기술 글에서는 1번과 2번을 우선 적용합니다.

- 좋음: `왜 coupon-issues SUCCESS는 발급 완료가 아닐까`
- 좋음: `쿠폰 발급도 접수와 처리를 분리해야 하는 이유, Kafka intake 설계에 대해`
- 좋음: `Redis Lua로 쿠폰 재고를 선점한 과정`
- 지양: `대규모 트래픽을 해결한 놀라운 비밀`
- 지양: 너무 추상적이거나 기술명 없는 제목

## 권장 소제목 패턴

- `서론`, `계기`, `과정`, `후기` 같은 익숙한 흐름을 허용합니다
- 질문형 소제목을 적절히 섞습니다
  - `왜 여기서 Kafka가 필요한가?`
  - `예상치 못한 예외는 없었을까?`
  - `운영에서는 무엇이 달라질까?`
- 본문 중 하나 정도는 긴장을 환기하는 제목을 허용합니다
  - `예상치 못한 병목 발견`
  - `헷갈리기 쉬운 지점`
  - `여기서 outbox를 오해하기 쉽습니다`

## 비유 사용 규칙

- 비유는 글당 1~2개 정도만 사용합니다
- 비유는 기술 설명의 보조 장치로만 씁니다
- 비유 직후에는 반드시 실제 구조나 동작 설명으로 돌아옵니다
- 억지로 웃기려는 문장, 밈, 과한 드립은 피합니다
- 최근 글처럼 가벼운 구어체를 살리더라도 본문 중심부는 반드시 기술적으로 정제합니다

## 안전한 비유 예시

- outbox:
  - `메인 업무가 끝난 뒤 영수증을 별도 보관함에 넣어 후속 처리를 맡기는 구조`
- Lua reserve:
  - `입구에서 번호표와 좌석 수를 동시에 확인하는 안내 데스크`
- Kafka intake:
  - `주문 접수와 실제 조리를 같은 테이블에서 하지 않는 구조`
- distributed lock:
  - `작업대에 동시에 두 사람이 올라오지 못하게 막는 안전 바`

## 금지 규칙

- `한다체` 로 계속 밀어붙이는 딱딱한 설명문
- `백엔드 개발자 관점에서` 같은 메타 설명이 반복되어 경험담의 온도를 죽이는 글
- 비유만 기억나고 기술 내용은 남지 않는 글
- 친근함을 살리겠다고 정확성을 희생하는 글
- 작성자 말투를 과장되게 흉내 내는 글
- 설명보다 감탄사와 수식어가 더 많은 글
- 제목이 모호해서 다 읽기 전까지 무엇을 설명하는지 보이지 않는 글
